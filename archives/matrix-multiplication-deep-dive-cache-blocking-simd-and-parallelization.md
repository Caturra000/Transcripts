## 1. 朴素实现 (Naive Implementation)

让我们从最简单的朴素矩阵乘法开始。它的逻辑非常简单：取矩阵 A 的每一行，乘以矩阵 B 的每一列，然后将结果放入矩阵 C 的对应元素中。这涉及到非常简单的三重循环（`i`, `j`, `k`）。这种实现我在 GitHub 的很多仓库里都见过。

**性能表现：**
执行时间需要 152 秒。在我的性能数据表中，我会展示执行时间、相对加速比（与上一个版本对比）、绝对加速比、执行的 GFLOPS，以及达到了峰值性能的百分之几。显然，这个版本的性能非常低。

## 2. 第一步优化：改变循环顺序 (Loop Reordering)

在朴素实现中，我们按照字母顺序迭代 `i`、`j` 和 `k`。但从数据访问模式（Data Access Pattern）来看，这是一种非常低效的方式。只需将循环顺序更改为 **`i`, `k`, `j`**，我们就能大幅提升算法性能。

**背后的原因：**
当我们以 `i, j, k` 顺序迭代时，我们按顺序访问矩阵 A 的元素，这很好；但我们访问矩阵 B 的元素时，**步长（Stride）非常大**，导致空间局部性（Spatial Locality）极差。
为什么大步长访问很糟糕？
1. **浪费缓存：** 当我们从内存中读取一个元素时，我们不仅仅读取那一个元素，而是读取相邻的几个元素来填满整个缓存行（Cache Line）。假设一次读 4 个元素，如果我们只用其中 1 个（跳到下一行了），我们就白白浪费了缓存内存。
2. **禁用硬件预取器：** 大步长迭代使得硬件预取器（Hardware Prefetcher）极难预测并将数据提前从内存提取到缓存中。

当我们将循环改为 `i, k, j` 时：
现在我们在最内层循环迭代 `j`，对矩阵 B 的访问变成了顺序访问（连续内存），这非常好。由于矩阵 A 所在的索引不依赖于 `j`，我们在内层循环中只是重复使用矩阵 A 的同一个元素，这也是极好的。同时，我们对矩阵 C 的写入也是顺序访问。

**性能表现：**
执行时间显示，它比朴素实现**快了 12.6 倍**！但相对于我们的理论峰值性能，它仍然相当慢。

## 3. 缓存分块优化 (Tiling)

下一步优化是分块（Tiling 或 Cache Blocking）。我们不再逐个元素相乘，而是将矩阵划分为一个个“块”（Block）。为了简化理解，假设一个块包含 4 个元素。分块算法计算的不再是矩阵 C 中的单个元素，而是 C 的整个块。

在这种情况下，我们重用的不再是矩阵 A 中的单个元素，而是整个块。通过这种技术，我们大幅增加了数据的重用率。然后我们迭代矩阵 B 的整个块行，来计算矩阵 C 的完整块。

在代码实现上，我们需要增加三个循环：最内层的循环将在块（Tile）内部迭代，最外层的循环将遍历各个块。我将最内层的循环提取到一个独立的函数中，称之为 **Kernel（微内核）**。

**性能表现：**
我们发现，使用分块优化后，性能达到了上一个版本（循环重排）的 **3 倍**。

## 4. 向量化 (Vectorization / SIMD)

简单说一下 SIMD：如果是标量加法，我们要把数组 A 和数组 B 的几个元素相加，需要执行 4 次独立操作。但在 SIMD 下，我们可以将所有这些元素加载到特殊的向量寄存器（Vector Registers）中，只需执行 1 条指令就能得到相同的结果。这就是我们要对矩阵乘法做的。

但在开始手动向量化之前，让我们先看看编译器生成的汇编代码。我们发现，**编译器其实已经为我们使用了 SIMD 指令**（它知道我们要干什么）。有趣的是我注意到一件事：当我把最内层循环移到单独的函数（Kernel）中时，汇编代码发生了巨大变化，且这个版本实际上跑得更快了。

现在，让我们用手动向量化的版本替换它，并与编译器自动向量化的结果进行对比。

**手动向量化 Kernel 实现：**
这个实现我最早是在 Ulrich Drepper 那篇著名的文章《每个程序员都应该知道的内存知识》中看到的（强烈推荐阅读）。但那篇文章比较老，使用的是 128 位 SIMD 指令；我的 CPU 支持 256 位，所以我使用现代一点的指令集（YMM 寄存器）对其进行了升级。

工作原理：
1. 取矩阵 A 的元素，将该值广播（Broadcast）填充到整个向量寄存器中（我们称之为 A 寄存器）。
2. 从矩阵 B 取出一整行（比如 4 个元素），加载到 B 寄存器中。
3. 从矩阵 C 加载元素到 C 寄存器。
4. 应用 SIMD 指令 `FMA`（融合乘加，Fuse Multiply-Add），同时执行乘法和加法。结果存入 C 寄存器。
5. 最后将结果存回矩阵 C 的内存中。

**对比编译器自动向量化与手动向量化：**
对比两者的汇编代码，我们发现：**编译器没有利用我的 CPU 所拥有的全部可用寄存器**。在我的 CPU 上有 16 个 YMM 寄存器，编译器只勉强用了 3 到 4 个；而在我的手动实现中，我成功用满了全部 16 个寄存器。

**性能表现：**
手动向量化比上一个纯分块版本快了 **1.23 倍**。

## 5. 缓存感知实现 (Cache-aware Implementation)

在深入此项优化前，简要回顾一下内存层次结构：金字塔顶端是最昂贵、最快、容量最小的内存（L1 缓存）；底端是硬盘（容量最大、最慢、最便宜）。为了解释如何利用这种层次结构，我引用了德克萨斯大学奥斯汀分校计算机科学教授 Robert van de Geijn 制作的一张图表。

看一眼我们微内核的第一层循环：当我们用 A 矩阵的一行去乘 B 矩阵的一列时，内核第一层循环在遍历矩阵 A 的各行。这意味着，我们正在**用矩阵 A 的所有这些行，来反复重用矩阵 B 的那个分块**。由于我们一直在重用 B 块，它是被最频繁访问的数据，所以它**极大概率会驻留在 L1 缓存中**。这给了我们一个估算 B 分块大小的依据：**B 分块的整体大小必须能装进 L1 缓存**。

在微内核的第二层外循环中，我们开始用矩阵 A 的一整个块，去乘以矩阵 B 的多个块。现在我们开始**重用矩阵 A 的整个块**了。这是次高频访问的数据，所以这部分应该放入 **L2 缓存**。知道了 L2 缓存的大小，我们就能估算 A 矩阵块的大小。

顺着这个逻辑，根据内存层次结构，我们可以为不同层级的内存估算不同的分块尺寸。

现在，我们不再只有 3 个最外层循环，而是增加到了 5 个，以利用所有的内存层级。最外层循环遍历大块，大块内遍历小块，小块内再由内核进行乘法。我用 `Nc, Mc, Kc` 表示 L1/L2 缓存块的大小，用 `Nr, Mr` 表示寄存器块的大小。

**代码影响：**
我们将固定大小的循环常量，替换为基于内存层次结构计算出的尺寸。但是当我们查看汇编输出时发现，使用这种技术后，**编译器又开始不好好利用全部寄存器了**！

**性能表现：**
尽管如此，我们还是看到了一点性能提升：比上一个版本快了 **1.16 倍**。但我们知道，我们还能做得更好。

## 6. 强制优化寄存器使用

我们要给编译器一个明确的提示：我们有多少 CPU 寄存器，并尝试用满它们。
我怎么做的？我在代码中手动创建了指定数量的 C 寄存器变量、B 寄存器变量和 A 寄存器变量。我的 CPU 共有 16 个 YMM 寄存器，所以我分配了 12 个寄存器给 C 矩阵元素，3 个给 B，1 个给 A（这个比例分配是从前面表现好的汇编代码里学来的）。

**性能表现：**
现在情况好多了，汇编显示我们用满了所有可用的 C 寄存器。这几乎让性能比上一个版本**翻倍**！

## 7. 多线程化 (Multi-threading)

我们有多个核心，为什么还不用呢？
对于这颗 CPU，我尝试了各种并行化循环的策略。我试过在顶层的嵌套循环处并行，试过把特定线程绑定到特定核心上……我尝试的一切几乎都给出了相同的结果，没看到什么明显的差异。

所以在这个案例中，我选择了最简单的方案：我只在行索引（`i` 维）上切分矩阵 C。每个线程处理这个矩阵的几行。

**性能表现：**
在我的 4 核 CPU 上，这直接带来了 **3 倍** 的性能飞跃。

## 8. 矩阵数据重排 (Matrix Packing / Data Layout)

让我们再次审视我们的微内核（Kernel），你会注意到我们在遍历矩阵 A 和 B 时，仍然存在**跨大步长**的情况。我们都知道这不太好。我们能在内核里把步长变成 1 吗？
答案是可以的。

怎么做？我没有在内核内直接按那种跳跃的模式访问数据，而是将这个访问过程**上移到内核外部**的顶层。我在外层先按内核需要的奇怪模式把数据连续地拷贝到一个 Buffer（缓冲区）里，然后再把这个 Buffer 传给内核。

这样一来，内核在遍历不同元素时就不再需要跳跃了，数据访问变成了绝对的顺序读取。代码实现上，原来那些巨大的步长变成了寄存器块的大小（比如 `Nr=12`, `Mr=4`）。这些数字非常小，编译器非常乐意为我们 **自动展开（Unroll）** 这些循环。在汇编里，微内核内最内层的嵌套循环彻底消失了，只剩下对 `k` 索引的一个单层循环。

**性能表现：**
这项优化结合前面的重排，给我们带来了 `1.17x` + `1.10x` 的加速。

## 10. 手动循环展开 (Manual Loop Unrolling)

我们刚才看到编译器为我们展开了最内层的循环，但我希望能**手动**展开它，看看是否对性能有影响。

问题在于，如果我们像写汇编一样手写展开的代码，我们就会失去改变寄存器数量的灵活性。比如，如果我把常量 `Nr` 改变了，代码就不会自动调整分配的 B 寄存器数量。

我们有没有办法在不手动把代码复制粘贴 N 遍的情况下生成展开的代码？有的。我使用了 **C++17 的折叠表达式（Fold Expression）和逗号运算符（Comma Operator）** 来在编译期生成展开的循环。

**技巧演示：**
假设有一个函数 `print()`，我们想调用它 4 次。我们可以写一个 `print_n_times`，接收一个参数包 `idx`，然后用折叠表达式展开这个调用。如果你需要调用 32 次甚至更多，手写参数包不现实，所以我们可以利用 C++ 标准库的 `std::make_index_sequence` 为我们生成这个序列。

我把这个技巧用在了我的 `compute_kernel` 中。通过传递索引包 `i` 和 `j`，我在编译期展开了 `i` 和 `j` 的循环。它会生成所需数量的 B 寄存器变量，并生成一长串 `compute_row` 的函数调用。
我们将这些代码包装进最终的接口中，只需要传入寄存器分块大小，它就会为我们生成完全展开的内核代码。我们将同样的技巧用在了把 C 寄存器写回内存的 `store_kernel` 函数上。

**这种写法的优点：**
这自动支持了所有其他数据类型，极易定制，我们可以通过传递不同的模板参数生成不同的内核。

**性能表现：**
对比编译器自动展开与我们手动模板展开的汇编，两者几乎一模一样，除了**一条指令的位置被改变了**（广播指令被挪到了循环的开头）。这有影响吗？有。它带来了 `1.02x` 的微小提升。不过说实话，很难去推断编译器是如何重排这些汇编指令的，试图去干预它更是难上加难，这次能捕捉到这个差异算是运气好。

## 与 OpenBLAS 的对比及性能分析的困境

当我把我的最终版本与权威数学库 **OpenBLAS** 进行对比时，发现 OpenBLAS 依然比我快了大约 **8% (1.08x)**。所以我的代码还有提升空间。

但是，当我们逼近峰值性能时，去分析为什么慢就变得极其困难，原因如下：

1. **看 C++ 源码已经没用了**：在这个阶段，我大部分时间都在看汇编输出。在汇编里分析某条指令为什么慢，得出的结论和我们在 C++ 里分析函数是完全不同的。比如，你在汇编里看到某条 FMA 指令是瓶颈，但修复它的方法可能是回到 C++ 去改变数据布局（Data Layout）。
2. **指令级并行 (ILP) 难以利用**：当我们在 C++ 里做修改时，很难给编译器暗示去做出最佳的指令重排。
3. **硬件 Bug**：我注意到一个极其离谱的案例：在你生成的汇编里，使用的寄存器不同都会影响性能。我坚信这是一个硬件 Bug。在这款 CPU 上，当代码使用 `RCX` 寄存器时，CPU 似乎无法利用 **寄存器重命名（Register Renaming）** 这项底层优化；而当某人把汇编里的 `RCX` 手动改成 `ECX` 时，性能就获得了提升！这种纯硬件级的微调，你是无法在 C++ 代码层面做到的。

**数据分析工具的矛盾：**
数据分析也不直观。我通常用 Perf 和 Intel VTune 来分析性能。
用 VTune 看，OpenBLAS 的各项微架构指标全面优于我的实现，这似乎能得出结论 OpenBLAS 更好。
但是换用 Perf 看，结论就不那么明显了：我的实现在**最后一级缓存（LLC）未命中率**上表现得比 OpenBLAS 更好，但 OpenBLAS 在 **IPC（每周期指令数）** 上表现更佳。光看这些零散的指标，很难判断谁的实际执行时间更短。

另一个例子：用 Perf 查看指令瓶颈，我的代码左侧出现了很多红色的长条（耗时多的指令），而修改后（比如把 K 循环也给展开了）红色警报消失了。直觉上性能应该变好了吧？但实际运行时间完全没变，我们只是把问题从分析工具的监控下“藏”起来了而已。

最诡异的例子是：我写了一个基于 128 位 SIMD 的非常慢的版本，但在 VTune 里面，它的微架构指标竟然比 OpenBLAS 还好看！但实际运行时间被 OpenBLAS 吊打。

## C++ 编写矩阵乘法的成效如何？

在研究这个课题时，我从 Kokkos、BLAS 等各种库以及 GPU 优化的文章中学到了很多。
我真的非常喜欢 C++ 提供的**零开销抽象**，尤其是 `templates` 和 `constexpr`。

**未来的力量：**
我目前正在试玩 **`std::mdspan`**。我最喜欢它的一点是，当我做矩阵重排、改变布局时，我现在可以**在编译期检查矩阵的维度**！这太棒了，因为手写矩阵乘法优化到后期时，你的大脑会因为各种索引计算而爆炸，你会忘记一些东西，此时你极度需要编译器的帮助。`mdspan` 做到了。
此外，`std::execution` 将能帮助我们更优雅地实现多线程策略。而对于那些根本不想自己写矩阵乘法的人，未来我们会有 `std::linalg`（线性代数标准库）。C++ 背后有着强大且活跃的社区支持。

**为什么很多 HPC 库（如 OpenBLAS）的内核是用纯汇编写的？**
他们大概是不想再和编译器做斗争了，特别是当你需要精准控制指令重排时。他们不想依赖于特定的编译器或版本。当我分别用 GCC 和 Clang 编译我的不同优化版本时，我观察到了巨大的性能差异（有时高达 40% 的波动）。不过好在，我的**最终优化版本**在 Clang 和 GCC 下表现完全一致。
还有就是应对前面提到的 `RCX` 和 `ECX` 寄存器导致的硬件 Bug，这种问题只有用汇编才能绕过。

## 换上现代硬件 (AMD Zen 5) 再次对决

直到这里，我似乎还没完全回答“C++ 是否足够高效”这个问题。

最近，我买了一台新的工作站，搭载了 **AMD Zen 5** 微架构（Ryzen 9 9950X）。我决定看看我的代码在现代 CPU 上的表现。
用同样的公式计算，它的理论峰值性能是 **2.9 TFLOPS**。但我查阅资料发现，如果执行极度繁重的 AVX-512 工作负载，该 CPU 会降频至 3.8 GHz。因此，在此时的最大有效峰值性能应该是 **1.95 TFLOPS**。

当我运行 OpenBLAS 和我的实现进行对比时，我极其惊讶：
**对于双精度浮点数（Double），我的纯 C++ 实现实际上击败了 OpenBLAS！**

我做了哪些调整？
1. 我微调了分块大小（Tile sizes），因为新 CPU 的缓存大小和寄存器数量不同了。
2. 我修改了多线程策略。不再只按行切分，而是按块（Block）切分。在这台 32 线程的系统上（其实是 16 物理核，我的代码还没针对超线程做针对性优化），我把矩阵划分成了 `4x8` 的网格让线程去处理。

刚才所有的实验都是针对双精度（Double）的。但是，**得益于 C++ 的模板，我们免费获得了单精度（Float）的支持！** 测试表明，对于单精度，我的实现与 OpenBLAS 性能打平。

**C++23 与 Bfloat16：**
在 C++23 中，我们引入了 `std::bfloat16`。既然模板是免费的，我决定直接把数据类型替换成 `bfloat16` 测试一下（它在 LLM 和 GPU 计算中非常常用）。
遗憾的是，编译器目前还无法为 `bfloat16` 自动生成高效的 AVX-512-BF16 汇编指令集。所以我写了几个模板特化（Template Specialization），手动调用了硬件指令。这只是临时的解决方案，未来编译器变聪明后就可以删掉。
我无法将其与 OpenBLAS 进行对比，因为 OpenBLAS 目前还不支持 `bfloat16`。最终，对于 `bfloat16`，我获得了 **3.09 TFLOPS** 的惊人性能。我除了换了个类型，基本上什么也没干！

## 总结

当我在最新的 Zen 5 上对比性能时，我发现 C++ 的表现完全处于顶级水准。因此，我坚信 **C++ 绝对能够交付与手写汇编相媲美的矩阵乘法性能**，而且我的代码甚至还不是最完美的，还有进一步提升的空间。

这就是我今天的全部分享，感谢大家的聆听！

## Q&A 问答环节

**Q1：你有跟 Intel MKL 库对比过吗？**

**A：** 大家都问我这个问题。实际上还有一个库叫 BLIS。我读过一篇基准测试文章，结果显示该库（包括 MKL）在 AMD CPU 上的性能下降非常严重。MKL 在 Intel CPU 上表现极佳，但在 AMD 上表现很糟糕。所以我这次在 AMD 上测试没有使用 MKL。

**Q2：我正在开发一个多维数组库，旨在提供你手动写的这些抽象（类似于更高级的 `mdspan`），我们可以会后交流一下。**

**A：** 没问题，非常期待交流！

**Q3：我注意到你提到内存是对齐的，但在底层 Intrinsics 代码里，我看到你使用了 `storeu` 和 `loadu`（未对齐版本）。**

**A：** 观察得非常仔细！实际上我尝试过使用强制对齐版本的指令（`_mm256_load_pd` 等），但我没有观察到任何性能差异，所以就留成那样了。

**Q4：关于使用 `mdspan`，你觉得在处理大量编译期维度时，需要什么特殊的抽象吗？标准库里是否缺失了某些功能？**

**A：** 对于如何构建完美的抽象我还没完全想好。我目前的进展是：用 `mdspan` 替换了原有的裸指针，这很好，完全内联，没有性能损失。然后我尝试自定义布局（Custom Layout）——也就是实现那个 Z 字形的跳转内存访问模式。但发现**编译器无法很好地优化自定义布局**，导致性能比预期差很多。目前我还在解决多线程下的矩阵边缘情况（不规则大小导致的尾部处理），等理清全局后，我可能会更清楚我们需要什么样的抽象。

**Q5：关于你的测试环境，你前面的测试说 152 秒，后面优化到了 1 秒内。但那是一台桌面级 CPU，还跑着图形界面，这会引起抖动（Jitter）。你是否应该在一个干净的 Linux 集群节点上运行测试？**

**A：** 你说得很对，我的测试环境不是“纯净”的，后台有各种进程。但是，我观察到一个现象：**高度优化的实现，通常不太受后台浏览器等进程的干扰**；反而是朴素的、未优化的实现，性能波动极大。另外，你提到的执行时间小于 1 秒导致计时误差的问题，确实存在。在最后测试极速版本时，我其实加大了矩阵的测试尺寸，以确保 GFLOPS 数据的准确性。

**Q6：在手动循环展开和内存预取时，你如何决定展开的次数和预取的距离？**

**A：** 对于预取距离：如果我正在使用当前元素，我预取的逻辑就是“抓取下一个缓存行”。对于矩阵 C，由于我没有把它的元素拷进连续的 Buffer 里，跨度很大，所以我用矩阵维度（大跨度）作为预取步长。关于展开次数：我只展开了两个最内层的 `i` 和 `j` 循环。我试过把外层的 `K` 也展开，但除了让 Perf 面板变好看之外，运行时间没有任何缩短。

**Q7：你尝试过不同版本的编译器吗？它们性能差异有多大？**

**A：** 一直在尝试。特别是在最后，因为我要用到 `mdspan`、`bfloat16` 和 `std::simd`，我必须换不同的编译器来满足这些前沿特性。目前我停留在 GCC，因为它支持全部特性。关于性能差异，在那些未达到最优的代码版本上，有时 GCC 比 Clang 强，有时 Clang 比 GCC 强，差异甚至能达到 **30%**。但在最终的极致优化版本上，两者几乎没有差异。

**Q8：你的缓存分块技术让我想起了图像处理中的卷积（Convolutions）。如果处理的是图像，你认为优化的思路和步骤会改变吗？**

**A：** 如果只挑其中一种技术（比如分块）应用进去，肯定是有帮助的。其实我今天展示的优化顺序，**并不是我开发时真正的推进顺序**，只是为了方便演示而排版的。你可以打乱这些优化技术的应用顺序，依然能看到性能提升。
*(另一位听众补充)*：卷积和矩阵乘法非常相似（比如 `im2col` 算法）。优化的思路（为了缓存而分块、为了线程而分块）是一样的。不同点在于你如何将卷积核加载进矩阵：卷积需要额外的数据搬运（数据复制重叠）。一旦转换好，后面的底层加速原理是相通的。

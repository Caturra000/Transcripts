## C/C++ 标准中的未定义行为（UB）

在标准中，有几种识别 UB 的方法。如果违反了“应（shall）”或“不应（shall not）”的要求，且该要求出现在约束条款（constraint clause）中，那就是未定义行为。C++ 有约束条款吗？是一回事吗？那只是 C 语言特有的吗？

*（观众互动：是的，我们有。但我们没有那样的规则。在 C++ 中，它实际上会直接写明“行为是未定义的”。大多数情况下我们是这样，大多数情况下我们会明确说“有符号整数溢出是未定义行为”。大多数情况下我们对用户关于未定义行为是诚实的，但在某些地方我们会撒谎。比如我们知道那里有未定义行为，但我们什么也不说。这主要涉及多线程。当我们把多线程加入标准时，我们掩盖了很多东西，因为那简直就是一场灾难。）*

标准中还有一种方式是通过“省略任何明确的未定义行为描述”来产生 UB。当我刚加入委员会时，我们还没真正讨论过“如果用 `INT_MIN` 除以 `-1` 取余数（`INT_MIN % -1`）会发生什么”。因此，这就成了“隐式的未定义行为”，因为我们没有明确说它是 UB。后来当我们意识到这一点时，我们添加了一些文本，使其成为显式的未定义行为。但这几种分类之间没有任何区别，侧重点也没有不同，它们都是未定义行为。

在 C 标准的附件 J.2 中，我们有一个未定义行为的列表。它由标准的编辑维护，列出了显式的未定义行为。显然不包括隐式的，因为我们根本不知道隐式 UB 都有哪些。但这个列表并不总是正确的，它虽然很方便、很好用，但有时会出错。它只是一个编辑性质的附件，所以没有受到和正文主体同样的严格审查。全凭编辑一个人完成。

当我最初加入 C 委员会时，我与两年前刚去世的 Tom Plum 博士交谈。他告诉我：“如果去参加奥运会，你在‘吹毛求疵（挑刺）’这个项目上拿了金牌，就会有人来找你，问你是否愿意加入 C 标准委员会。”所以那里的多数人都非常精于此道，他们真的很享受挑刺。

那么，为什么有些东西会被归为未定义行为呢？很多人对未定义行为存在深深的误解。最常见的误解是认为“它是标准中的错误”。这完全不对。我们在把它们写进去时花了很多心思，它们存在都是有原因的，只是这些原因通常不为大家所知。

第一个原因，是赋予实现者（编译器开发者）一种“豁免权”，使他们不必去捕获某些难以诊断的程序错误。大家都知道有“停机问题”，有些问题在计算上是不可行被识别出来的。所以必须要有这种宽容度。而且，C 委员会里的人对彼此都非常好。假设有这样一个讨论：“谁能诊断出这个行为？”除了 Bob，其他人都举手了。他们会对 Bob 说：“Bob，你是不是还在考虑怎么诊断它？”Bob 会说：“不是，我们的实现架构做不到诊断这个。”大家就会说：“好吧，别担心，我们把它定为未定义行为。”所以他们互相非常友善。但这并不意味着 11 个编译器中有 10 个不会去诊断它。这只是意味着他们**不被强制要求**去诊断。因此，除非你对你所使用的特定编译器有足够了解，否则你不能依赖这种诊断。

第二个原因，是为了定义一些模糊的边界情况（corner cases），避免偏袒某一种实现策略。有些硬件具有特定的特征，而这些特征并不总是一致的。比如有符号整数溢出或除法，在 x86 Intel 芯片上的除法指令总是会触发陷阱（trap），而在 PowerPC 上，它会静默产生一个未定义的结果。左移操作在不同计算机上也是不同的。如果你把一个 32 位的数字左移 32 位，其实并没有一个逻辑上绝对正确的答案，这根本无关紧要，这种操作本身就很愚蠢。正因为如此，开发硬件的人也不知道答案应该是什么，所以他们就自己编造行为，这就导致了不同的硬件行为。大多数人不想做的事情就是，强迫编程语言为一种荒谬的计算提供一个可预测的结果。因为那样你会毫无理由地拖慢程序的运行速度。所以这也是 UB 的一个类别，基本上就是“让硬件做硬件该做的事”。作为程序员，你必须容忍存在一系列可能的行为，不过大多数情况下这并不重要，因为你谈论的是一个本就不该存在的古怪边界情况。

最后，是为了标记一些可能的、符合标准的语言扩展点，让实现者可以增强语言。我最近在委员会里和人讨论过这个问题。有时他们说，所有未定义行为都是这样，它们都是实现的扩展点。我最常谈论的例子是 `fopen`。有一组特定的模式字符，如果你使用了不同的模式字符，那就是未定义行为。现在，假设你使用了字符 `y`，它是未定义的。并不会有任何东西爆炸，我非常怀疑是否有任何库实现会因为这个多出来的字符而崩溃。他们这样做的原因是，标准只能在广泛平台共有的文件系统行为子集上进行标准化。C 语言甚至不承认“目录（directories）”的存在，这是一个标准之外的概念，更不用说文件权限之类的东西了。所以这些额外的模式字符让实现者能够在目标平台上为你提供对特定文件系统特性的访问权限。这非常有用，绝对是一个扩展点，它不是那种会让程序爆炸或因为你用了 `y` 就会被编译器胡乱优化的东西。但这种行为仍然被 MISRA（汽车工业软件可靠性协会）所禁止，因为 MISRA 不理解 UB 的工作原理以及它的设计初衷。现场有 MISRA 的人吗？Andreas 在。Andreas 是的。通常情况下，我会对在场的人说点好话，Andreas 懂得其实比他们多，但他放任这些人这么做。而我，正如他可以作证的，就只会和所有人干仗。

## 编译器处理 UB 的策略与模型

当（编译器）实现发现未定义行为时，它可以完全忽略它，产生不可预测的结果。在 C 委员会中，我们常说“程序可能会跑去玩《生命游戏》（Game of Life）”，因为 C 委员会里的人年纪都够大了，我们还记得当年《生命游戏》是计算机上的大应用。在 C++ 里，我想我听过“鼻涕恶魔（nasal demons）会从你鼻子里飞出来”这种说法。他们（编译器）可以以一种具有该环境特征的有据可查的方式表现。再说一次，比如 Intel 上的 `INT_MIN % -1`，会导致中断向量 0 发生错误。这就是该环境的特征。它不优雅，但这就是那个环境的做法。或者在最好的情况下，它会终止翻译或执行，并给出一个诊断信息。

对于 C 或 C++ 编译器来说，优化器的基本设计与其他任何语言相同。其基本原则是**用一种更有效、但计算结果相同的方法来替换原有的计算**。但有些优化会改变行为。接下来是我在这个演讲中要说的可能最愚蠢的一句话：有时优化实际上会消除未定义行为。这些是“好的”优化。而有时优化会引入安全漏洞，这些是“坏的”。

C 语言有一个 "As-if"（如同）规则，它规定了抽象机器上计算的结果，但同样没有指定方法。因此，编译器被允许使用任何它想要的方法来产生相同的结果。在抽象机器中，所有的表达式都会按照语义规定被求值。我见过人们犯的一个错误是：如果你想弄清楚你的 C 或 C++ 程序到底在做什么，不要去想优化。只思考抽象语义，按照抽象语义一步一步走。因为编译器**被要求**产生相同的结果，即使它在得出结果的过程中进行了优化或做了不同的事情。

实际的实现如果发现表达式某部分的值没有被使用，或者它没有提供任何必要的副作用（包括调用函数或通过 volatile 访问对象），就不需要对这部分进行求值。有些函数是纯函数（pure functions），不会产生副作用。但很多时候编译器并不知道，特别是如果它位于不同的翻译单元（translation unit）中。所以很多时候编译器会采取简化的假设，假定函数调用会产生副作用。再强调一次，编译器可以选择产生正确结果的任何方法。

这就赋予了编译器在构建程序时移除被视为未使用或不需要的代码的余地，这通常是有益的。安全领域的人，他们只学到了一点皮毛，而没有了解全部，基于这点皮毛，他们开始给出糟糕的建议，比如“关掉你所有的优化”。实际上，优化通常会让你的代码运行得更快，有时是快得多。而且有时它们会让你的代码变得更好。所以关闭优化通常不是一个好方案，但这往往是人们在得知某些优化可能在代码中引入漏洞时的膝跳反射（本能反应）。

安全和 UB 的问题尤其在于：很多 UB 可以成为漏洞，比如缓冲区溢出，越界写入对象，这是未定义行为，也是大家公认的漏洞。程序员（尤其是安全人员）可能会这么做：他们会写代码说“如果是未定义行为，就把这个当成错误情况处理”。但问题是，**“检查某事是否是未定义行为”的测试代码本身，往往就是未定义行为**。如果你在代码中问“这里有 UB 吗？”，你在提问的过程中可能就已经触发了 UB。随后，你的测试代码就被优化掉了。你看着你的代码，心想“我正在测试所有这些错误条件（这些 UB）”，但实际上并没有，因为它们被编译器移除了。所以你在提出“这里是否有 UB”的问题时必须非常小心，你得确保你在提问时本身没有触发 UB。

编译器有几种实现策略：

1. **硬件行为（Hardware Behavior）**：在这里，你的编译器只是生成汇编代码，汇编代码做什么就是什么。这在过去许多年里几乎是普遍的策略，所以许多早期的 C 和 C++ 程序员认为这才是正道。这也是为什么我在幻灯片上放了曼达洛人的头盔（台词：“This is the way”）。对于我们在座的老年人来说，这是我们从小到大习惯的模式。我们自然而然地假设编译器会生成一些指令，这些指令该干嘛干嘛。但这越来越不符合现实了，这让那些内化了“硬件行为模型”的人感到非常困扰和不信任，比如“指针出处（pointer provenance）”。使用相同的指针，进行位模式比较完全相同，但编译器却认为它们不是同一个指针。内化了硬件模型的人无法理解怎么会这样。
2. **超级调试模式（Super Debug Mode）**：你可以提供一个密集的调试环境，捕获几乎所有的未定义行为。这会严重降低性能，因此很少用于部署应用程序。像 Address Sanitizer (ASan) 和 UB Sanitizer (UBSan) 做的就是这种事。
3. **完全许可模型（Total License Model）**：编译器将任何可能的 UB 视为“不可能发生的情况（can't happen condition）”。这允许非常激进的优化。

当我说“编译器有某种实现策略或模型”时，我其实在撒谎。编译器作者真正做的是：在他们想这么做的时候，随意混用这些模型，毫无一致性可言。非常相似的代码结构，可能在这个地方被使用硬件行为模型处理，在另一个地方又被使用完全许可模型处理。大多数编译器供应商的目标是让他们的编译器能成功编译一些源代码，并且不想破坏原有的代码。编译器作者也不一定是坏人，他们是在努力适应编译器的用户。但这无助于代码的移植性。

## 常量折叠与基础优化

一个非常简单的优化叫做“常量折叠（Constant Folding）”。这里有几行代码，非常简单，没做什么复杂的事：

```c
/* 生成代码，仔细甄别 */

int cst = 17;
int x = 11 - 7;
// 其它使用了 cst 和 x 的代码...
```

我们如何简化它呢？首先，我们其实不需要分配那个常量变量的寄存器或内存位置。我们可以直接删除那行代码，将该值直接替换到指令中。所以不用 `cst`，直接用 `17`。接下来是 `11 - 7`。这是一个常量表达式。同样，在编译期间它可以被编译器进行常量折叠。`11 - 17` 直接被替换为 `-6`。现在我们已经大大简化了这段代码。这会自动为你发生，无论在何种优化级别，甚至是完全未优化的构建（`-O0`）中。你大多数情况下无法关闭它。这个优化就是会发生。

某些语言特性在前端可能就需要这个。编译器即使在未优化的构建（为了产生可用的调试二进制文件）中，也会运行一些微不足道的优化。我一直提到的例子是 `INT_MIN % -1` 评估结果是什么？最初在 C 语言中这是隐式的未定义行为，后来变成了显式的。不同的编译器会有不同的结果：

```c
/* 生成代码，仔细甄别 */

int result = INT_MIN % -1;
```

如果我们在 x86_64 GCC 下用 `-O0` 编译，结果是 `0`；用 `-O3` 编译，结果也是 `0`。这里发生的事情是它被编译器进行了常量折叠，编译器可能使用了一个更大的类型（比如 `intmax_t`，最大的有符号类型）来计算，所以能得出结果 `0`。如果你只看了这个，你可能会认为这段代码绝对没问题，因为“嘿，我得到了 0，0 是数学上正确的答案”。但在 x86_64 Clang 上，你得不到 `0`，你会得到那种典型的 UB 垃圾值。大多数人会期望这两种编译器得出相同的结果，但事实并非如此。

这两种结果都不算“错”，因为它就是 UB。任何事情都可能发生。编译器也可以在检测到 UB 时直接拒绝编译。比如 Microsoft Visual Studio 会直接报错说“存在溢出的常量除法”，然后停止编译。你必须先修复它才能继续。这也是被允许的，因为它是 UB。一旦编译器检测到 UB，他们想怎么做就怎么做，通常他们是想做一些有用的事情，也就是本演讲的主题——优化你的代码。

你需要小心这些简单的测试，因为不同的编译器对 UB 产生的效果不同，而且测试本身可能没有考虑到预期之外的优化。所以，当我写这样的小测试时，我通常会查看生成的汇编代码，以确保生成了我以为的那些指令。这就是我最初 20 年写 C 语言代码的方式。我学 C 语言的方法就是写点代码，编译它，看看汇编，然后通常我会想：“嗯，这和我想的不一样。”然后我就去读读标准，试着去预测。当我达到基本能预测会生成什么指令的水平时，我就觉得自己懂得了足够多，可以站在大家面前谈论这些东西了。

## 指针算术与指针溢出

好的，现在进入本场演讲的重点：指针和整数相加。

在 C 语言中，当一个整数类型的表达式加到一个指针上，或者从指针减去时，结果拥有该指针操作数的类型。如果是 `int*`，结果也是 `int*`；如果是 `float*`，结果也是 `float*`。任何使用数组记法（`array[i]`）的表达式，最终都只是被翻译为带有比例因子的指针算术运算（scaled pointer arithmetic）。使用数组索引写法并不能让你摆脱指针算术运算，它们本质上是一样的。

如果指针操作数和运算结果没有指向同一个数组对象中的元素，或者没有指向该数组对象最后一个元素之后的一个位置（one past the last element），又或者加减法产生了溢出，那么行为就是未定义的。这里的溢出与“有符号整数溢出”不同，这是一种处理指针与整数相加时的特定溢出，对于通常不会溢出的无符号整数来说，这也适用。

假设这里有一个数组，你可以形成指向每个元素的指针。你还可以形成一个指向数组之后一个位置（too far）的指针。这是允许的，因为标准化之前的许多早期 C 代码都是不断递增这个指针直到它达到“too far”的位置，然后用等值比较来终止循环。ANSI C 委员会不想破坏所有那些旧代码。所以他们说，“是的，我们允许形成这个 too far 的指针。”但是，你不能解引用这个“too far”指针，那是 UB。并且你不能再次递增它，一旦你再次递增它，就触发了 UB。遗憾的是，这种 UB 很少被检测到，如果能检测到就好了。显然，如果指针发生了回绕（wraparound）或溢出，那也是 UB，只是另一种类型的 UB。这并不高深。

C++ 的情况也是一样的。这只是 C++ 标准里的说法，可能现在已经变了，因为 C++ 委员会那些家伙都不睡觉的，他们总是在改东西。他们有一个比我们大得多的委员会。

关于指针算术的快速问答：

```c
/* 生成代码，仔细甄别 */

int len = /* some external input */;
int *pointer = /* ... */;
auto res = pointer + len; 
```

`res` 的类型是什么？没错，是 `int*`。

那么如果是这样呢：

```c
/* 生成代码，仔细甄别 */

int len = 0;
int *pointer = NULL;
auto res = pointer + len; 
```

在 C++ 里它的求值结果是什么？是 `nullptr`。在 C++ 中，如果你把一个 null 加上 0，你得到 null；把 0 加上 null，得到 null；从 null 减去 0，得到 null。**但是，如果你把 null 加上任何非零值，你会得到未定义行为**。所以上面因为赋了 `0` 和 `NULL`，在所有情况你都会得到 null。不过，直到 C23 之前，C 语言的行为是不同的，我认为在这些情况下都会直接得到 UB。从 C2Y（当前的标准工作草案）开始，我们采用了一种更接近 C++ 的行为。这是一篇由 Aaron Ballman 和一个好像是 Intel 的人提交的提案。

最后一个测试：我们有一个长度为 5 的整数数组 `A`。我们有一个 `int j`，一个指向数组的指针，以及计算结果 `res = pointer + j`。那么 `pointer` 和 `j` 指向哪里？如果 `A + j` 在 `0` 到 `too far` 指针的范围内，它就指向 `A[i + j]`。**否则，你就会得到未定义行为**。减法同样适用：如果 `pointer - j` 越界了，就是未定义行为。

一旦有了未定义行为，不要指望它在数学上还能讲得通。它可以是任何东西。在那种情况下，试图去推理程序的行为已经不再可能了。这就是为什么一个带有 UB 的程序不可能被认为是安全或可靠的，因为你无法推理它的行为。

## 边界检查漏洞与“完全许可”模型

有些关心安全性的程序员可能会想：“我听说缓冲区溢出是个很糟糕的事情。让我试着做一些边界检查来防止它。”

```c
/* 生成代码，仔细甄别 */

// size_t 类型的 len
if (pointer + len > max_pointer) {
    // 错误处理
}
```

（顺便说一句，用 `size_t`，即无符号整数类型来表示大小。关于为什么使用 `ssize_t` 来表示大小是一种极端的厌恶，我能做一小时的专门演讲，更多人应该听听那个讲座，因为很多 C++ 开发者在这个问题上都偏离了正轨。当然，我不是那种固执己见的人...笑）

所以我们做了这样一个测试。如果 `pointer + len` 大于 `max`，我们就将其作为错误处理，因为显然这超出了对象的边界。但是不管你采取哪种模型（硬件行为模型还是完全许可模型），这段代码中都有一个 Bug：对于非常大的 `len` 值，`pointer + len` 会发生溢出。这就是我几张幻灯片前谈到的那种溢出（指针加上极大偏移量导致地址回绕），而不是算术溢出，从而导致 UB。

在硬件行为模型下，结果通常会回绕，指向内存中比 `pointer` 地址更低的地方。所以一个有经验的、内化了硬件行为模型的程序员可能会写出下面这样的边界检查：

```c
/* 生成代码，仔细甄别 */

if (pointer + len < pointer || pointer + len > max_pointer) {
    // 错误处理：捕获了回绕和越界
}
```

这是老练的程序员的做法。当你试图向 C 委员会提出争论时，你可能会说：“程序员经常犯这种错误。”但一般而言，C 委员会根本不在乎经验不足的程序员。他们真的不在乎，一点都不在乎。他们非常关心性能。所以，如果你找到他们说：“我有一个很棒的安全解决方案，它会增加 5% 的开销。”基本上，他们会一脚把你踢出门外，想都别想。如果你对他们说：“我有一个很棒的安全解决方案，它能提升 1% 的性能。”他们马上洗耳恭听，很想知道具体细节。如果你对他们说：“我有个零开销的安全方案。”他们中的一些人仍然会反对，因为他们不想在安全问题上浪费时间。这就是现实。但通常零开销的方案还是能推销出去的。如果是 1% 的开销，你就得下大功夫了。所以，在过去的 20 年里，安全性并没有因为标准的推进而必然变得更好。

*（观众席有人提问/打断，麦克风声音）*

*观众：显然，如果结果超出了数组末尾，`pointer + len` 就是未定义行为。所以跟其他任何东西都无关。当你写下 `pointer + len` 的那一刻，如果 `len` 太大，你已经越界了。*

*我：是的，这是对的。如果在边界外，这就是 UB。你的意思是，在做比较之前，它已经是 UB 了对吗？*

*观众：你似乎认为如果 `len` 太大，它会回绕。我是在说它是否太长或者是否回绕根本不重要。在你在两件事之间进行比较之前，你已经有了未定义行为。回头看你上一张幻灯片。*

我：是的，你完全正确。程序员写下那种用来试图保护自己不越界的代码，实际上是在**显式地写下未定义行为**，而不是让它隐式地发生。你是对的，这也就是我想说的重点，只不过我本来打算花更多时间绕个圈子来说它。但这确实是一个常见的错误。

这个特定的错误是在贝尔实验室的 Plan 9 团队中出现的。那些都是非常有经验、留着灰白胡子的资深大佬。虽然这个检查（`pointer + len < pointer`）看起来能起作用，但它在分段地址体系上会失败。标准委员会指出，溢出的指针是未定义的，这就允许实现者（编译器）假设**在任何架构上，指针溢出都永远不会发生**。

因此，遵循 **完全许可模型（Total License Model）** 的编译器可以直接消除 `pointer + len < pointer` 这个子表达式。这又导致整个边界检查失效，因为我们又回到了 Bug 的最初版本。这正是刚才那位观众指出的，这也是被允许的，因为计算 `pointer + len` 时已经发生了 UB，编译器可以假设这个 UB 永不发生。结果就是，这段比较代码变成了死代码（Dead Code），被直接删除。现在情况甚至比刚开始还要糟糕，因为它看起来像是有测试，但实际上根本没有执行，而且编译器也不会给你诊断警告。

其实情况比我准备这个演讲时还要糟糕。我本来打算说，如果你使用 `-Wstrict-overflow=3` 选项，它会诊断出这个问题。当然，如果你没听过我的演讲，没人会想到去用这个选项。但事实上，这个补丁/优化是由 Google 的一个家伙写的。幸运的是我记不起他的名字了，不然我现在就会脱口而出。

在他制造了这个问题之后，我在 CERT 写了一份相关的漏洞通报（稍后会展示）。我们开始与他沟通，他说：“哦，我会添加一个诊断信息。”但结果证明，他添加那个诊断信息只是为了让我们闭嘴。他根本没打算真正修复这个问题，他不认为这是一个问题。所以他的解决方案就是对我们撒谎，让我们别再烦他了，这事我也是几年前才发现的。这生态系统可真是太可爱了，真的。

在完全许可模型下，优化的过程是这样的：编译器面对 `p + V1` 和 `p + V2` 的比较（`p` 是同一个指针，`V1` 和 `V2` 都是整数值）。模型允许这被简化为仅仅比较 `V1` 和 `V2`。然而，如果 `V1` 或 `V2` 的大小使得它们与 `p` 相加时产生溢出，那么简化后的比较并不能给出与“先计算再比较”相同的结果。通常计算机算术并不总是遵循数学的代数恒等式。这种转换之所以被允许，唯一的理由就是其行为是未定义的。编译器可以忽略这种情况发生的可能性，并生成在数学层面上结果不正确的代码。

回到我们的例子：

`pointer + len < pointer` 就等同于 `pointer + len < pointer + 0`。这里是对同一个指针加上两个整数进行的比较。它可以被简化为比较这两个整数：`len < 0`。由于 `len` 是无符号类型，它小于 0 当然是不可能的，所以这段判断条件永远为假。因此，没有任何理由不把这个测试从代码中完全删掉。

一旦你意识到了这个问题（要意识到并不容易，因为没有诊断信息，你只能通过测试用例来发现），它是很容易补救的。你可以这么写：

```c
/* 生成代码，仔细甄别 */

if (pointer <= max && len <= (max - pointer)) {
    // 安全操作
}
```

因为我们知道 `pointer <= max`，所以在 `max - pointer` 的计算中没有回绕或溢出的可能性，这种检查就永远不会被编译器移除。解决方案其实很简单，问题在于开发者压根不知道存在这个问题。

## 编译器标志、诊断与修复建议

这很有趣，因为这一切都发生在互联网时代，你可以在网上搜到这事。我们在 CERT 调查了这个由 Plan 9 报告的漏洞。我们在 GCC 开发者邮件列表中发帖说这里有个漏洞，然后人们用各种难听的话骂我们。你可以谷歌一下，看看他们是怎么骂我们的。他们会说：“我们被允许这么做（因为这是UB）！”我们会回答：“是是是，我们知道这是 UB，你们可以这么做，但问题是，你们**应该**这么做吗？”那个其实是态度最“好”的 Google 开发者，后来证明只是用公关手段哄骗我们闭嘴的。

邮件列表上有人告诉我们：

“`buf + n` 对于很大的 `n` 值可能会回绕，导致像这个例子一样的未定义行为。如果将其优化为 `n < 100`，代码会好得多，你就消除了未定义行为，有了可预测的执行结果。”

这是他们指出的例子：

```c
/* 生成代码，仔细甄别 */

// 假设 buf 是数组首地址
if (buf + n < buf + 100)
```

如果你用 GCC（这里参考的 2008 年的版本，所以近 17 年来你用的任何 GCC 都会这么做）编译，它会把它直接简化为 `n < 100`。所以，一个非常好的建议是：**不要依赖编译器来为你进行安全性的优化**。因为优化不是强制要求，它是被允许的，但不是必须的。

为了消除这种未定义行为，你应该手动进行优化，直接把代码改成 `n < 100`。大多数聪明的开发者可能不会直接写出那种 `buf + n < buf + 100` 的代码，但这种代码大量来源于宏展开。你传入 `buf` 和 `n` 以及 `100`，你并没有意识到它会展开成带有 UB 的操作。这就很棘手了。你只能通过把代码送入预处理器，检查预处理后的代码来发现这种事，或者依赖工具（前面说过，编译器通常不诊断这个问题）。

GCC 后来在一个维护版本（Maintenance Release）中改变了这个行为。也就是说，你只是打算升级到一个修复 bug 的版本，嘭！你的未定义行为产生的效果全变了！所以，UB 实际上是一个代码可移植性的问题，而且非常致命：它甚至可能在同一个编译器的不同发布版本之间发生改变。

所以我们在 2008 年写了一份漏洞通报（CERT Vulnerability Note）。2015 年我在马德里的一个开源会议上提到这个，有个跟我交谈的人惊呼：“天哪！你就是写那个高危漏洞通报的人！”在某一部分极客圈子里，我最出名的居然是这个，他们可能永远不知道我写了 7 本书，但他们知道我在 2008 年发了那个警告。而且据我所知，这也是 CERT 发布的唯一一个此类漏洞通报。通常 CERT 只通报在已经部署的代码中发现的实际漏洞，而这个不是部署的漏洞，这只是一个关于优化的极其恶劣的问题，它让你极其容易地部署出一个漏洞。GCC 的人对我们非常愤怒，而 Microsoft 的人则对我们感激涕零。

关于编译器的标志：

这种优化在 `-O2` 及以上级别（包括 `-Os`）执行。在 `-O1` 或 `-O0` 不执行。过去，可以通过设定 `-Wstrict-overflow=3`（或更大）来检测，或者用 `-fno-strict-overflow` 来禁用。但就像我说的，这只是为了让我们闭嘴。到了 GCC 第 8 版，他们停止维护这个标志了。现在它时好时坏，而且会产生大量的误报（false positives）。所以我们不得不决定停止推荐这个标志。现在已经没有静态诊断这个问题的原生方案了，推荐的做法是使用 UBSAN（Undefine Behavior Sanitizer）来追踪。但那毕竟是动态分析，你必须得实际走到那个代码路径才能发现问题。这么多年过去了，这依然是个大问题。

`-fstrict-overflow` 允许编译器假设“严格的有符号溢出规则”，即采用完全许可策略，假设有符号算术溢出是 UB 且不发生。这里要注意，如果你足够细心，会发现他们的文档用词并不准确。他们把“指针溢出”和“有符号算术溢出”混为一谈了。这是两种不同的溢出，但文档中用同一个术语把它们打包处理了。这会让懂行的人感到困惑。

当严格溢出生效时，像 `i + 10 > i` 这样的判断，对于有符号数 `i`，编译器会直接将其优化为永远为 `true`。大多数情况它是 true，除非加法发生了回绕或溢出。因此，在评估此类优化时，必须极其小心地编写测试代码，确保测试本身不包含溢出，否则测试会被直接优化掉。更恶心的是，这默认在 `-O2` 开启。这意味着在 `-O0` 下做调试测试时，程序的行为和最终发货的版本行为是不同的。所以我总是认为这是一个寻找 UB 的好方法：同时构建 `-O0` 和 `-O3`，如果行为不一样，那肯定是有 UB，去把它揪出来。

安全界的人通常会告诉你使用 `-fwrapv`。它指示 GCC 假设加减乘等有符号算术溢出会发生回绕（它不影响除法或取余）。这类似于 `-fno-strict-overflow`。这曾经为 Java 的前端启用（Java 规定有符号整数溢出回绕），后来 Java 前端没了，但这个选项还在。但这存在一个性能与安全权衡的论点。优化通常能消除很多 UB，提升代码性能。如果你关注安全性，提升的性能可以留出更多 CPU 周期去进行额外的输入验证和测试。性能好对于安全来说并不完全是坏事。

关于 Clang，这很有趣。Clang 拥有和 GCC 一模一样的所有标志。但是，很多时候这些标志只是为了让原本写给 GCC 的构建脚本能在 Clang 上运行而不报错。所以实际上在 Clang 里，其中很多标志根本不起任何作用，完全被忽略。不过，Clang 的设计在我看来更好，它不需要开启优化级别就能提供诊断警告，而 GCC 经常把诊断逻辑塞在优化器代码里，你在 `-O0` 编译时可能根本看不到一堆警告。因此，在开发构建周期中，你应该在一个开启全面诊断，并且优化级别设置为你要出货的级别（比如 `-O2`）的环境下进行测试。这对于保障代码安全至关重要。

到了 GCC 8，`strict-overflow` 被弃用。`-fno-strict-overflow` 映射到 `-fwrapv` 和 `-fwrapv-pointer`。现在在所有优化级别下，有符号整数溢出默认都是 UB。使用有符号整数溢出检查器（Sanitizer）是目前推荐的代码审计方式（配合模糊测试 Fuzzing）。

## 有符号整数溢出与循环不变式外提

我们进一步探讨有符号整数溢出。它是 C 标准中的未定义行为。实现可以静默回绕（最常见的），也可以抛出陷阱（Trap），或是二者的结合。前面问过大家关于 Intel 处理器的行为，其实正确答案是：加减乘会静默回绕，但除法和取余操作会触发陷阱。这就是混合处理的例子。

有人问过是否有一种实现会产生饱和（Saturation）操作。既然它是 UB，如果你写了一个实现去做饱和操作，那是完全允许的。今天早上我收到 UB 研究组主席的一封邮件，他恰好用有符号整数溢出举例，说可以把任何 UB 看作一个扩展点。比如这允许饱和运算。但他也说，在它作为 UB 存在的 50 年里，没有任何人使用过这个扩展。这让人怀疑我们是否真的需要这个没被用过的扩展点。编译器作者现在绝对不肯放弃它，因为这会严重拖慢现有代码的运行速度。甚至像 IBM XL 这样的编译器，默认将**无符号**整数回绕也视为“不可能发生”的未定义行为！这虽然不符合标准，但用户习惯了。这也说明符合标准有时没你想的那么重要。

来看这段代码：

```c
/* 生成代码，仔细甄别 */

int j = 0;
for (int i = 1; i > 0; i += i) {
    j++;
}
```

这段代码假设：如果一个正数不断翻倍，最终会变成负数（溢出回绕），循环就会终止。但由于它是有符号整数，我们面临 UB。在较新版本的 GCC 和 `-O2` 下，它采用完全许可模型，假定溢出不可能发生，因此认为这个数可以永远翻倍且永远变大，这段代码会被编译成一个真正的无限死循环。GNU C 库中曾经就犯过类似的错误（在处理 `time_t` 类型时），不过很久以前就修复了。

接下来讲“循环不变式外提（Loop Invariant Hoisting）”。

循环不变式代码是指可以在不影响程序语义的情况下移出循环体的语句。基本上就是循环执行期间永远不会改变的代码。如果把它们提到循环外，通常就能只执行一次，从而提升性能。

阅读 C 标准时，每当它给你提供某种“保证”时，你应该把那些话当成圣旨供起来，因为它很少这么做，大多数时候它都在说“任何事都可能发生”。标准里有这样一句话：**检测到有符号整数溢出的实现，不能将不会产生整数溢出的程序转换为会产生整数溢出的程序。**

注意“检测到（detect）”这个词。以 Intel 为例，除法和取余会抛出陷阱，这算是“检测到”。而加减乘的回绕不算是检测到。因此标准的真正意思是：**我们不会把一个原本没有崩溃的程序，优化成一个会崩溃的程序**（但如果是回绕，我们就忽略不管了）。

例子：

```c
/* 生成代码，仔细甄别 */

// 假设在一个检测有符号溢出的虚构架构上
if (z < INT_MAX) {
    int x = z + 1; // 绝对安全
}

// 编译器不能将其转换为：
int x = z + 1;     // 如果 z 是 INT_MAX，这里就会触发溢出崩溃
if (z < INT_MAX) {
    // ...
}
```

编译器不能这么做，除非它能通过值传播证明 `z` 绝对不会导致溢出。

再看一个更复杂的代码。我花了不少心思才让编译器按照我想要的方式进行优化。这里有一个非常昂贵的除法操作 `si1 % si2`，且这两个变量在循环中不改变。

```c
/* 生成代码，仔细甄别 */

// 伪代码演示
while (/* 循环条件 */) {
    // 很多代码...
    log_message_2();
    int result = si1 % si2; // 循环不变式，存在除零或 INT_MIN % -1 的潜在风险
    // 很多代码...
}
```

编译器垂涎三尺，迫不及待想把这个昂贵的操作提到循环外面。在未优化的情况下，如果输入导致除法陷阱（比如除以 0 或 `INT_MIN % -1`），程序会按顺序输出 `log_message_1`（循环前），然后是 `log_message_2`（循环内），然后程序因为陷阱崩溃。

但是在微软 Visual Studio 下开启优化后，这个带有潜伏 UB 的除法操作被整体提到了循环开始之前。结果是：你看到了 `log_message_1`，然后程序就直接崩溃了！`log_message_2` 再也没有机会执行。所以这里的问题是： **UB 可以在多大程度上发生时间上的位移？** 当时 Intel 的专家 Clark Nelson 的意见是：“对于程序的输出，它是可以被追溯的（retroactive），因此把注定要发生的 UB 向上提升，越过标准 I/O 的调用是被允许的。”当时没人质疑这个说法，现在的 UB 研究小组正在重新审视这类问题。

Intel 的专家甚至问过：“你能把 UB 提权（hoist）到一个 `volatile` 访问之前吗？”当时我说：“我们的对话到此为止吧，越聊越吓人了。”因为在编译器作者问出这种问题时，就说明他实际上已经这么干了，他只是在想办法让它合法化。

## 循环归纳变量优化

最后十分钟。另一种优化叫“循环归纳变量（Loop Induction Variables）”。

归纳变量是指在每次循环迭代中以固定数值递增/递减的变量，或者是另一个归纳变量的线性函数。

```c
/* 生成代码，仔细甄别 */

for (int i = 0; i < n; i++) {
    int j = 17 * i; 
    // ...
}
```

这里 `i` 是归纳变量。`j` 是 `i` 的线性函数，也是归纳变量。编译器可以用更简单的计算来替换它，因为加法比乘法便宜。编译器会把它重写成：

```c
/* 生成代码，仔细甄别 */

int j = 0;
for (int i = 0; i < n; i++) {
    // 使用 j 进行操作...
    j += 17; 
}
```

这把昂贵的乘法变成了每次迭代的加法，大幅提升性能。由于循环归纳优化经常涉及到有符号溢出（前面提到 IBM XL 也处理无符号回绕）。

问题在于，如果原代码中有一个循环：

```c
/* 生成代码，仔细甄别 */

for (int i = low; i < high; i++) {
    int sum = i * 53; 
    // ...
}
```

编译器可能将其优化为：

```c
/* 生成代码，仔细甄别 */

int IC = low * 53;
int MAX_VAL = high * 53;
for (/* 新的基于 IC 的迭代 */) {
    // 使用 IC...
    IC += 53;
}
```

这种优化的危险在于它并没有提供完全相同的行为。如果在传入的参数中，`high * 53` 在执行循环前就立即导致了溢出呢？原代码可能循环根本不会执行，所以原始乘法并不会发生。但在优化后的代码中，循环外预先执行的乘法触发了 UB 或者回绕，导致计算出了一个很小的负数值，这可能会彻底改变循环的执行次数和逻辑。

这种被修改后的代码，只有在**不存在溢出**的前提下，其行为才与原代码一致。这也就是“完全许可模型”的定义：假设不存在未定义行为。如果你的数据触发了溢出，那就出大问题了。而且极其难以调试，因为那些发生溢出的变量是由编译器自动生成的辅助变量，源代码里根本没有，你甚至不知道它们的存在。

C 和 C++ 规定无符号整数采用模算术（回绕）。GCC 团队说他们绝不会假设无符号整数不回绕来进行优化。但 IBM XL 编译器默认连无符号回绕也优化掉了。它有一个 `strict-induction` 选项来关闭这种可能改变程序语义的归纳变量优化。

## 指针出处（Pointer Provenance）与别名分析

最后，我跳过关于“严格别名（Strict Aliasing）”的部分，直接谈谈一些比较新的东西：指针出处（Pointer Provenance）。

这也是我们 C 委员会刚刚完成的一个 TS（技术规范，TS 6010），应该一两天后就会正式发布。指针出处问题起源于大约 2000 年向委员会提交的缺陷报告，因为编译器厂商希望通过 **别名分析（Alias Analysis）** 来做更好的优化。这就引发了关于他们能做什么的许多未解问题，导致不同编译器对指针出处的处理大相径庭，且互不兼容。我们推出的规范注定会破坏一部分现有代码，因为没有任何一种解决方案能让现有的所有实现都正常运行。

看下面这个例子：

```c
/* 生成代码，仔细甄别 */

int x = 1;
int y = 2;
int *p = &x + 1; // 合法，指向 too far 位置
int *q = &y;     // 假设在内存中 y 刚好紧随 x 之后

// 使用 memcmp 防止编译器优化掉比较
if (memcmp(&p, &q, sizeof(p)) == 0) {
    *p = 11; // 给 p 指向的内存写入 11
    printf("x=%d, y=%d, *p=%d, *q=%d\n", x, y, *p, *q);
}
```

这段代码的输出是什么？

*   **GCC**: `1, 2, 11, 2`
*   **Intel**: `1, 2, 11, 11`
*   **Clang**: `1, 11, 11, 11`

三个编译器给出了三个截然不同的结果！原因在于它们采用了不同的 C 语言编译和运行模型。

**1. 具体模型（Concrete Model / 硬件模型）**

编译器生成对应的汇编，让硬件决定行为。指针就是指向内存的物理地址。在 Clang 中，`p` 和 `q` 在物理上指向相同的内存位置（即 `y` 的位置）。当你执行 `*p = 11`，这就物理地修改了内存，所以 `y` 和 `*q` 都会变成 11。这符合老派程序员的直觉。

**2. 抽象模型（Abstract Model）**

这也是带来高级优化（基于别名分析）的模型。语言类型强制执行严格的区别。编译器会推理指针的**出处（Provenance）**：它来自哪里？GCC 使用这种抽象模型。在 GCC 看来：

*   `p` 来自于 `x`，它只是一个指向 `x` 的（太远位置的）指针。
*   `q` 来自于 `y`。

当 GCC 看到 `*p = 11` 时，它认定这是在修改与 `x` 相关的对象（尽管实际上解引用 too-far 指针本身是 UB）。当随后代码要打印 `y` 或 `*q` 时，GCC 会反向推理：“程序有没有修改过 `y`？没有（因为 `p` 在逻辑上只与 `x` 有关，不可能修改 `y`）。所以可以直接使用之前 `y` 的值（2）。” 因此，写入 `*p` 的操作并没有反映在 `y` 上，这就导致了 `1, 2, 11, 2` 这个极其反直觉的输出结果。

像 C 委员会这样的组织非常保守，我们不想凭空发明规则，所以我们把它放进了一个 TS（技术规范 TS 6010）里，给实现者机会去测试和反馈，然后再把它加入到下一版本的正式标准中。
